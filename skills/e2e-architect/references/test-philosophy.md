# Test Philosophy — What Makes a Test Business-Aware

This document defines the quality bar. Every test generated by e2e-architect
must pass these criteria. Read this BEFORE writing any test code.

## The Five Principles

### 1. Test Names Are User Stories

The test name should read like something a product manager would write.
A developer reading only the test names should understand what the product does.

| Quality | Example |
|---|---|
| **Bad** | `"should render the project editor page"` |
| **Bad** | `"test navigation to dashboard"` |
| **Bad** | `"project CRUD operations"` |
| **Good** | `"new admin can create their first project during onboarding"` |
| **Good** | `"viewer can browse a published project and filter by tags"` |
| **Good** | `"free plan user sees upgrade prompt when hitting the project limit"` |

**Pattern:** `"[persona] can [business action] [context/condition]"`

### 2. Assertions Validate Business Outcomes

Don't assert that elements exist. Assert that business goals are achieved.

| Quality | Example |
|---|---|
| **Shallow** | `expect(page.getByRole('button', { name: 'Publish' })).toBeVisible()` |
| **Shallow** | `expect(page.getByText('Project created')).toBeVisible()` |
| **Business-aware** | After publishing: navigate to the public URL as a different browser context (viewer) and verify the project content is accessible |
| **Business-aware** | After creating a workspace: verify it appears in the admin's dashboard AND is accessible via the team's workspace list |

**The acid test:** If your assertion would pass even when the feature is broken
(e.g., a toast appears but the data wasn't saved), it's shallow.

### 3. Tests Follow Persona Journeys

Don't organize tests by page. Organize by what the persona is trying to accomplish.

**Anti-pattern — page-by-page:**
```
describe('Login page', () => { ... })
describe('Dashboard page', () => { ... })
describe('Project editor page', () => { ... })
```

**Correct — journey-based:**
```
describe('Admin onboarding journey', () => {
  test('new admin signs up and creates their first workspace', ...);
  test('new admin adds a project with sections and tasks', ...);
  test('new admin publishes and verifies the public URL', ...);
});
```

Each `describe` block is a journey. Each `test` is a meaningful step within
that journey. Steps may share state (sequential) or be independent (parallel),
depending on the journey structure.

### 4. Business Rules Are First-Class Assertions

Don't just test the happy path. Test the constraints and limits that define
the product's business model.

| Business Rule | Test |
|---|---|
| Free plan: max 1 workspace | "free plan user cannot create a second workspace" |
| Published project requires at least one item | "admin cannot publish an empty project" |
| Only admin can edit their workspace | "unauthenticated user gets redirected from workspace settings" |
| Subscription required for premium | "expired subscription blocks access to premium features" |

### 5. Edge Cases Test Failure Modes That Matter

Don't test every possible error. Test failures that affect the business:

| Matters | Doesn't matter |
|---|---|
| Suspended account shows reactivation path | 404 page has correct styling |
| Payment failure shows retry option | Form field has red border on error |
| Session expiry redirects to login cleanly | Loading spinner appears during fetch |

## Anti-Patterns — Tests This Skill Must NEVER Generate

### The Visibility Test
```typescript
// BAD — asserts element exists, not that the feature works
test('project editor has save button', async ({ page }) => {
  await page.goto('/projects/1/edit');
  await expect(page.getByRole('button', { name: 'Save' })).toBeVisible();
});
```

### The Smoke Test Disguised as E2E
```typescript
// BAD — navigates to a page and asserts it loaded
test('dashboard loads', async ({ page }) => {
  await page.goto('/dashboard');
  await expect(page).toHaveTitle(/Dashboard/);
});
```

### The Framework Test
```typescript
// BAD — tests that React renders, not that the product works
test('project list renders items', async ({ page }) => {
  await page.goto('/projects');
  const items = await page.getByTestId('project-item').count();
  expect(items).toBeGreaterThan(0);
});
```

### The Hardcoded Wait
```typescript
// BAD — uses arbitrary timeout instead of web-first assertion
test('order is processed', async ({ page }) => {
  await page.click('#submit-order');
  await page.waitForTimeout(3000);
  await expect(page.getByText('Order confirmed')).toBeVisible();
});
```

## Quality Checklist

Before finalizing any generated test file, verify:

- [ ] Every `test()` name reads like a user story
- [ ] Every assertion validates a business outcome (not just DOM state)
- [ ] Tests are grouped by journey, not by page
- [ ] At least one business rule is tested per journey (if rules apply)
- [ ] No `waitForTimeout` — only web-first assertions
- [ ] No CSS selectors — only accessible locators (`getByRole`, `getByLabel`, `getByText`)
- [ ] No `toBeVisible()` without a business context (it's OK as part of a larger assertion chain)
- [ ] Test file has a JSDoc header explaining which business flow it covers
